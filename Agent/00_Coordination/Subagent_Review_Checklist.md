# 子代理实施与审查规范

生成日期：2026-07-14
用途：给 ClaudeCode 或其他子代理说明“实施报告必须写什么”，也给 Codex 说明“实施后必须怎么审查并反馈给用户”。

## 0. 子代理报告最低要求

子代理每次实施完成后，必须用中文报告：

- 实际修改了哪些文件，是否超出 Prompt 范围。
- 新增 / 修改的 C++ 类名、父类、职责和文件路径。
- 成员变量名、类型、UPROPERTY 暴露方式和作用。
- 函数名、参数、返回值、UFUNCTION 暴露方式和作用。
- 是否修改 Build.cs、配置、蓝图、地图或资产。
- 是否新增 GameplayTag、Attribute、DataAsset、GA、GE、GameplayCue。
- 是否编译，编译命令和结果。
- 需要用户人工在 UE 编辑器里做什么。
- 如何验证才算成功。

如果无法用 MCP / 自动化完成 UE 编辑器操作，子代理必须停止自动尝试，改为输出人工操作清单。

## 1. 范围检查

- 是否读取了 Prompt 要求的文件。
- 是否只修改 Prompt 允许的文件。
- 是否有额外改动。
- 是否误改 UE 资产、蓝图、地图、配置。
- 是否使用了禁止使用的 MCP / 编辑器操作。

## 2. C++ 命名审阅表

Codex 反馈给用户时必须包含：

| 项 | 必填 |
| --- | --- |
| 新增类名 | 是 |
| 父类 | 是 |
| 文件路径 | 是 |
| 类职责 | 是 |
| 成员变量名 / 类型 / 作用 | 是 |
| 函数名 / 参数 / 作用 | 是 |
| UPROPERTY / UFUNCTION 暴露原因 | 视情况 |
| 是否新增 GameplayTag | 是 |
| 是否新增 Attribute | 是 |
| 是否新增 DataAsset / GA / GE / Cue | 是 |

## 3. 设计检查

- 类职责是否单一。
- 是否符合当前分层。
- 是否有过早抽象。
- 是否把玩家输入、相机、技能逻辑放错层。
- 是否有学习型注释解释关键设计原因。
- 是否避免噪音注释。

## 4. 编译与验证检查

- 是否编译通过。
- 如果未编译，原因是什么。
- 如果只完成代码，是否明确“未 PIE 验证”。
- 如果涉及 UE 资产，是否编辑器可见、可打开、可保存。
- 如果涉及运行时，是否给出单人 PIE 验证步骤。
- 如果涉及多人，是否给出 Listen Server + Client 验证步骤。

## 5. 人工 UE 操作清单

如果需要用户进入 UE 编辑器，Codex 必须输出：

1. 打开什么路径。
2. 创建什么资产。
3. 父类选什么。
4. 细节面板填什么字段。
5. 保存/编译哪里。
6. PIE 怎么验证。
7. 看到什么才算成功。

## 6. Git 建议

Codex 必须说明：

- 当前是否建议提交。
- 文档、C++、资产是否应分开提交。
- 是否需要先验证再提交。
- 是否需要 tag。
- 是否有用户/子代理未确认改动不能纳入本次提交。
