# 技能机制分类讨论稿

*创建日期：2026-06-26*  
*状态：讨论稿，不是最终设计。*

## 目的

我们要先回答“到底要支持哪些技能机制”，再决定代码怎么写。

这里的“机制”不是具体技能名字，而是技能背后的结构。例如：
- 加血和加蓝都属于“属性修改/资源恢复”。
- 火球和箭矢都属于“投射物”。
- 灼烧和中毒都属于“持续伤害 Debuff”。
- 蓄力射击和蓄力法术都属于“按住蓄力、松开释放”。

## 第一层分类：按生命周期

| 类型 | 例子 | 需要验证的系统能力 |
| --- | --- | --- |
| 瞬发 | 治疗、直接伤害、驱散 | 成本、冷却、GE 应用、失败原因 |
| 前摇/施法 | 火球、重击 | Montage、Commit 时机、取消窗口 |
| 蓄力 | 蓄力射击、蓄力法术 | 按住/松开、蓄力曲线、资源预留 |
| 持续引导 | 光束、持续治疗 | 周期消耗、目标刷新、取消 |
| 开关型 | 光环、持续护盾 | Infinite GE、开关状态、清理 |
| 被动 | 受击反弹、击杀回蓝 | 事件监听、内部冷却、防递归 |
| 多阶段 | 三段连招、二段跳斩 | 阶段状态、输入窗口、动画窗口 |

## 第二层分类：按目标选择

| 类型 | 例子 | 需要验证的系统能力 |
| --- | --- | --- |
| 自身 | 自疗、护盾、开关光环 | Self Target、GE to Self |
| 单体 | 点名治疗、锁定伤害 | 目标验证、阵营过滤 |
| 地面点 | 暴风雪、火墙 | 鼠标选点、地面验证、范围提示 |
| 范围 | 震荡波、AOE 治疗 | Sphere/Box/Cone 查询 |
| 投射物路径 | 火球、箭矢 | Spawn、命中回调、EffectContext |
| 近战 Trace | 普攻、连招 | Montage 窗口、Trace、HitResult |
| 链式目标 | 链式闪电 | 候选排序、防重复、最大链数 |

## 第三层分类：按效果

| 类型 | 例子 | GAS 承担什么 | 我们要补什么 |
| --- | --- | --- | --- |
| 直接伤害/治疗 | 火球伤害、治疗术 | GE / ExecCalc / Attribute | 目标和参数组织 |
| 持续伤害/治疗 | 灼烧、中毒、HOT | Duration/Period GE | 应用时机和表现 |
| Buff/Debuff | 加速、减速、虚弱 | GE、Tag、Stack | 配置和 UI 查询 |
| 护盾 | 魔法盾 | Attribute 或 GE | 伤害路由策略 |
| 控制 | 眩晕、沉默、定身 | Tag、GE Duration | 行为阻断规则 |
| 召唤/陷阱 | 召唤物、地雷 | 部分 GE/Tag | Actor 生命周期和所有权 |
| 位移 | 冲刺、传送、击退 | 部分 AbilityTask/CMC | 移动验证和碰撞 |

## 第四层分类：按表现

| 类型 | 例子 | 注意点 |
| --- | --- | --- |
| 动画 | 施法、近战挥砍 | Notify 只能调度，不单独决定权威伤害 |
| Niagara | 火球轨迹、爆炸 | GameplayCue 负责表现 |
| 音效 | 施法音、命中音 | 可走 Cue 或表现配置 |
| UI | 冷却、资源不足、范围提示 | 查询服务应和激活规则一致 |
| 镜头/震动 | 大招、命中反馈 | 纯表现，不改玩法状态 |

## 建议第一批原型

第一轮建议只做 6 个，避免范围失控。基于用户反馈，当前边界是：
- 先偏 ARPG，不在第一批主动引入 FPS / MOBA / 卡牌构筑类专门机制。
- 多人能力要纳入设计和验证，但实施顺序可以先保证本地玩家技能链路，再做 Listen Server / 多客户端验证。
- 第一批不做位移技能，例如冲刺、传送、击退。
- 召唤和陷阱重要，因为它们会验证 Actor 生命周期、所有权和清理，但不进入第一批，先放到第二批候选。
- 允许正式实施前做轻量资产盘点；具体技能资产仍在写具体技能前确认。

1. **瞬发治疗/伤害**
   - 验证技能配置资产、成本、冷却、GE 应用。

2. **火球投射物**
   - 验证投射物 Spawn、命中结算、GameplayCue、现有代码迁移。

3. **范围持续地面技能**
   - 验证地面目标、范围查询、持续效果、范围表现。

4. **蓄力射击**
   - 验证按住/松开、蓄力曲线、取消、资源预留。

5. **三段连招**
   - 验证 Montage、输入窗口、命中窗口、近战 Trace。

6. **受击反弹被动**
   - 验证被动事件、内部冷却、防递归。

## 第二批原型候选

- 持续光束。
- 链式闪电。
- 开关光环。
- 陷阱。
- 召唤物。
- Hitscan。

## 已确认事项

1. 第一批原型偏 ARPG。
2. 技能系统需要考虑多人，但不要求一开始就直接跳到复杂 Dedicated Server 验证。
3. 第一批不做位移类技能。
4. 第一批采用当前 6 个原型。
5. 召唤和陷阱不进入第一批，先放到第二批候选。
6. 准许在正式实施前做轻量资产盘点；具体技能资产仍在写具体技能前确认。
