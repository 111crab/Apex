# V1-A 技能实施顺序

*创建日期：2026-06-26*  
*状态：节奏确认稿。具体代码实现必须等运行时模型和单技能 Prompt 确认后开始。*

## 核心规则

V1-A 的技能不作为一个大包一次性实现，而是一个一个单独实现、单独验证、单独审查。

原因：
- 每个技能验证的系统能力不同，混在一起会让问题难定位。
- 用户需要学习 UE C++ / GAS，每个技能都应该有清晰讲解和注释。
- 多人验证需要逐个技能确认表现、状态同步和权威结算。
- Claude 的任务范围应该小而明确，方便 Codex 审查。

## V1-A 技能顺序

| 顺序 | 技能 | 主要验证能力 |
| --- | --- | --- |
| 1 | 火球术 | 投射物、命中、伤害、GameplayCue、服务器权威 |
| 2 | 复苏脉冲 | 治疗、自身/友方目标、HoT、状态同步 |
| 3 | 烈焰领域 | 地面选点、区域 Actor、周期伤害、持续表现 |
| 4 | 蓄力贯穿箭 | 按住/松开、蓄力倍率、释放时机、预测边界 |
| 5 | 三段裂刃 | Montage、近战 Trace、输入窗口、连招阶段 |
| 6 | 荆棘护盾 | 护盾、受击监听、反弹、内部冷却、防递归 |

这个顺序可以在架构设计后微调，但原则上先做基础传递和效果，再做输入时序、动画时序和被动反应。

## 每个技能的固定流程

1. **单技能设计说明**  
   写清楚技能玩法、输入、目标、效果、表现、多人验证点。

2. **资产需求确认**  
   列出需要的动画、Niagara、音效、图标或占位资产。用户确认后再实施。

3. **Claude 实施 Prompt**  
   Codex 写单技能 Prompt，限定允许改动范围、验证方式、注释要求和报告路径。

4. **代码与资产配置实施**  
   Claude 按 Prompt 实施，必要时使用 UE MCP。

5. **单机验证**  
   先确认本地玩家可以激活、命中、结算、显示反馈。

6. **Listen Server / 两客户端验证**  
   验证其他端表现是否正常、应同步状态是否正确、对其他角色的影响是否由权威端正确应用。

7. **实施报告与审查**  
   Claude 写报告，Codex 审查代码、文档、同步逻辑和学习型注释。

8. **用户确认**  
   用户确认后再进入下一个技能。

## 验证标准

每个技能至少要回答：
- 技能是否能被正确授予和激活。
- 成本、冷却和失败反馈是否正确。
- 目标选择是否符合设计。
- 玩法结果是否由权威端结算。
- 其他客户端是否能看到合理表现。
- 相关 Attribute / GameplayEffect / GameplayTag 是否同步到该同步的端。
- 取消、死亡、目标失效、Actor 生命周期是否有清理。
- 代码注释是否解释了关键 GAS 生命周期和设计取舍。
