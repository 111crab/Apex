# 技能机制分类确认稿

*最后更新：2026-06-26*  
*状态：Phase 1 当前确认稿。后续若用户重新调整范围，需要同步更新本文件。*

## 目的

本文件确认 Aura 技能系统第一批要覆盖的技能机制范围。它不是具体代码设计，也不是技能编辑器设计；它用于指导下一步 `GAS_Architecture_Options.md`，帮助我们判断 V1 运行时架构必须支持哪些能力。

## 第一批方向

第一批技能原型偏 ARPG，先验证通用技能系统的基础能力，不提前引入 FPS、MOBA、卡牌构筑等专门机制。

第一批暂不做：
- 位移类技能，例如冲刺、传送、击退。
- 召唤和陷阱。它们很重要，但会提前引入持久 Actor 生命周期、所有权、清理和复制复杂度，先放到第二批。

## 第一批 6 个原型

| 原型 | 主要验证能力 | GAS / UE 重点 |
| --- | --- | --- |
| 瞬发治疗/伤害 | 成本、冷却、目标、GE 应用、失败反馈 | `CommitAbility`、Instant GE、Attribute 修改 |
| 火球投射物 | 投射物生成、命中结算、表现同步 | Projectile Actor、HitResult、EffectContext、GameplayCue |
| 范围持续地面技能 | 地面选点、范围查询、持续效果、范围表现 | TargetData、Duration/Periodic GE、Area Actor 或查询服务 |
| 蓄力射击 | 按住/松开、蓄力曲线、取消、资源处理 | AbilityTask、输入事件、Commit 时机、预测边界 |
| 三段连招 | 动画窗口、输入窗口、近战命中、阶段推进 | Montage、GameplayEvent、Trace、Combo State |
| 受击反弹被动 | 被动监听、内部冷却、防递归 | GameplayEvent/Attribute 事件、Tag Gate、Cooldown GE |

## 多人验证最低标准

第一批技能不要求一开始就进入完整 Dedicated Server 验证，但不能按纯单机思路设计。

最低标准：
- 本地玩家技能链路先可用。
- 使用 Listen Server / 两客户端验证关键技能。
- 其他端能看到正确表现，例如施法、投射物、命中、范围效果、持续效果。
- 应同步的状态能正确同步，例如生命/法力、冷却、Buff/Debuff、控制状态、可见 GameplayCue。
- 对其他角色的影响必须由权威端正确应用，例如伤害、治疗、持续伤害、被动反弹。
- 纯表现可以本地预测或由 GameplayCue 驱动，但不能反过来决定权威玩法结果。

## 资产盘点原则

允许在正式实施前做轻量资产盘点。盘点目标不是马上选定所有最终资产，而是回答：
- 第一批每个原型需要哪些资产类型。
- 项目现有教程资产、UE 内置资产、占位资产里是否有可用候选。
- 哪些技能可以先用占位表现，哪些技能必须至少有动画、Niagara 或音效才能验证。

具体到某个技能正式实施时，再列出它需要的动画、Niagara、音效、图标或占位资产，并让用户确认。

## 第二批候选

第二批可以考虑：
- 持续光束。
- 链式闪电。
- 开关光环。
- 陷阱。
- 召唤物。
- Hitscan。

召唤和陷阱虽然不在第一批，但仍然是重要候选，因为它们能验证持久 Actor 生命周期、归属关系、死亡/过期清理、目标过滤和复制策略。

## 对下一步架构设计的要求

下一步 `GAS_Architecture_Options.md` 至少要解释：
- 为什么 V1 不能简单走“每个技能一个完全独立 GA 类”。
- “少量通用 GA + DataAsset 配置”能覆盖第一批哪些能力，不能覆盖哪些能力。
- 哪些逻辑必须保留 C++ 原生实现，以保证多人、预测、性能和可维护性。
- 哪些内容可以作为数据配置，为未来技能编辑器服务。
- 第一批原型会如何映射到 GA、GE、GameplayTag、GameplayCue、AbilityTask、Actor、DataAsset。
