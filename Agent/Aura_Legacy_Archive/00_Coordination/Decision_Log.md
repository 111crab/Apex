# 决策记录

*最后更新：2026-07-07*

这里记录长期有效的项目决策。未来 agent 不应该随意推翻这些决策，除非有新的明确原因。

## 2026-06-26 - 建立 Agent 文档区

**决策：** 使用 `Agent/` 存放协作文档，使用 `.agents/ue-project-context.md` 存放 UE skill 需要读取的项目上下文。

**原因：** 项目需要让用户、Codex、Claude 都能长期追踪规划、交接、审查和阶段总结。

**影响：**
- 游戏代码仍放在 `Source/`。
- UE 资产仍放在 `Content/`。
- Prompt、报告、规划、决策写入 `Agent/`。
- UE skills 优先读取 `.agents/ue-project-context.md`。

## 2026-06-26 - Codex 规划，Claude 实施

**决策：** Codex 负责高层规划和审查，Claude 负责按 Prompt 执行具体实现。

**原因：** 这样能把架构/审查和代码执行分开，减少误改和范围失控。

**影响：**
- Codex 的 Claude Prompt 写入 `Agent/04_AgentHandoff/Claude_Prompts/`。
- Claude 的实施报告写入 `Agent/04_AgentHandoff/Claude_Reports/`。
- 重大实现由用户审批方向。

## 2026-06-26 - 用户审阅文档优先中文

**决策：** 需要用户审核、讨论或决策的文档尽量使用中文。

**原因：** 用户明确提出该偏好，中文文档也更适合当前项目讨论。

**影响：**
- `Agent/` 下的规划、审查、报告优先中文。
- `.agents/ue-project-context.md` 这类工具上下文可以保持结构化英文；如果需要用户审阅，补中文说明。

## 2026-06-26 - 正式代码必须服务学习

**决策：** 后续正式实现代码时，必须写有价值的学习型注释，并同步补充中文设计/实现文档。

**原因：** 用户希望通过项目理解 UE C++、GAS、解耦设计、网络同步和编辑器扩展，而不是只得到最终代码。

**影响：**
- Claude 实施 Prompt 必须显式要求注释和中文实现说明。
- Codex 审查代码时，要把“注释是否解释了 GAS 生命周期和设计原因”纳入验收。
- 注释应解释关键机制、边界和权衡，不写显而易见的语法注释。

## 2026-06-26 - 第一批技能原型采用 ARPG 基础机制

**决策：** 第一批技能原型采用 6 个 ARPG 向基础机制：瞬发治疗/伤害、火球投射物、范围持续地面技能、蓄力射击、三段连招、受击反弹被动。

**原因：** 这组原型覆盖 GAS 技能系统最核心的几类问题：属性变化、投射物、范围与持续效果、输入时序、动画命中窗口、被动触发。它们能支撑后续架构讨论，又不会过早引入召唤、陷阱、位移等复杂生命周期问题。

**影响：**
- 第一批不包含位移类技能。
- 第一批不包含召唤和陷阱，它们进入第二批候选。
- 第一批需要考虑多人正确性：其他端应能看到正确表现，同步应有状态，且对其他角色的影响应能正确应用。
- 允许在正式实施前做轻量资产盘点，但具体技能资产选择仍在写具体技能前确认。

## 2026-06-26 - V1-A 具体技能逐个实施和验证

**决策：** V1-A 具体采用 6 个代表性技能：火球术、复苏脉冲、烈焰领域、蓄力贯穿箭、三段裂刃、荆棘护盾。每个技能必须单独设计、单独实现、单独验证、单独审查。

**原因：** 这些技能足够像正常游戏技能，又能覆盖不同系统压力。逐个实施能降低复杂度，方便学习、调试、多人验证和代码审查。

**影响：**
- Claude 的实施 Prompt 应一次只覆盖一个技能。
- 每个技能实施前需要列出资产需求并让用户确认。
- 每个技能至少完成本地玩家验证和 Listen Server / 两客户端关键同步验证。
- V1-A 完成前，不展开链雷、符文标记、哨卫图腾、影步突袭。

## 2026-06-26 - 技能机制大类目录已确认

**决策：** `Agent/02_SkillSystem/Skill_Mechanic_Catalog.md` 中的技能机制大类当前已确认，作为架构设计和未来技能编辑器模块划分的参考目录。

**原因：** 当前目录已经覆盖主要市面技能机制，继续无限细分会降低讨论效率。后续如果在实现或参考其他游戏时发现遗漏，可以增量补充。

**影响：**
- Phase 1 可以收束，进入 Phase 2 运行时架构设计。
- V1-A 不需要等待机制目录继续扩展。
- 机制目录服务于架构和编辑器，不直接等于实现清单。

## 2026-06-26 - V1 运行时采用克制版方案 C

**决策：** V1 运行时架构采用“克制版方案 C”：少量通用 GA + DataAsset 配置 + 少量原生机制片段。暂不做完整 SkillGraph / IR / VM。

**原因：** 该方案能让技能编辑器未来保存配置资产，同时保留 GAS 生命周期、多人与关键机制的 C++ 原生控制。它比每个技能一个 GA 更可复用，也比第一版直接做完整图系统更稳。

**影响：**
- 火球术等技能优先通过通用主动 GA 读取配置资产执行。
- 投射物、范围、蓄力、连招、被动监听等高风险机制以原生机制片段实现。
- 技能配置资产记录技能 Tag、输入、成本、冷却、目标、Delivery、Effect、Presentation 等数据。
- 下一步先写 `Skill_Runtime_Model.md`，之后做轻量资产盘点，再写第一份 Claude 实施 Prompt。
- 在单技能 Prompt 被用户确认前，不进入 C++ 实施。

## 2026-07-07 - V1 运行时模型主路径确认

**决策：** V1 配置技能主路径采用 `UAuraSkillDefinition + UAuraConfiguredActiveAbility + AbilitySpec.SourceObject`。技能数据保存在配置资产中，通用 GA 激活时从 AbilitySpec 的 SourceObject 取回具体技能定义。

**原因：** 该路径能复用少量通用 GA，同时保留每个技能独立的数据、输入、成本、冷却、目标、Delivery、Effect 和 Presentation 配置。它也直接服务未来技能编辑器：编辑器修改配置资产，而不是为每个技能生成新的 C++ 类。

**影响：**
- 第一份 Claude 实施 Prompt 只围绕火球术最小闭环实现该路径。
- 旧教程 GA 路径暂时保留，新配置技能路径并行接入。
- 后续技能编辑器优先编辑 `UAuraSkillDefinition` 类资产。

## 2026-07-07 - 保存旧教程代码基线

**决策：** 在进入 V1 技能运行时代码改写前，通过 Git tag `tutorial-baseline-before-v1-skill-runtime` 保存当前教程基线，并推送到远端。

**原因：** 用户希望后续可以在其他位置拉取最原始的教程版本作为对照，避免重构后失去学习项目原貌。

**影响：**
- 后续可以自由改写或迁移旧教程代码，但需要解释保留、迁移或替代的理由。
- 不因为已有 tag 就粗暴删除旧代码；V1 先保留旧路径并新增配置路径。

## 2026-07-07 - 技能资产优先复用 Aura 教程资产并倾向软引用

**决策：** 每个具体技能实施前先分析资产需求，优先在 `Content/Assets` 中寻找可复用的 Aura 教程资产；不够时再考虑 UE 官方内容、简单占位表现，或请用户补充外部资产。配置资产中的表现资源默认倾向软引用。

**原因：** 教程资产可以降低第一批原型的表现成本，软引用可以避免未来技能配置资产数量增加后造成不必要的加载膨胀。

**影响：**
- Claude Prompt 必须包含资产查找范围和软引用倾向。
- 如因 V1-A 快速验证而使用硬引用/同步加载，代码注释或文档必须说明这是阶段性折中。

## 2026-07-07 - 允许直接添加 UE C++ 文件

**决策：** 后续可直接在 `Source/Aura/Public` 与 `Source/Aura/Private` 添加 `.h` / `.cpp` 文件，不强制通过 UE 编辑器“添加 C++ 类”向导。

**原因：** 直接添加文件是 UE C++ 项目中的正常开发方式。只要反射宏、`.generated.h`、模块依赖和文件路径正确，UE Header Tool、IDE 和编辑器都能识别。

**影响：**
- 新增反射类型必须遵守 UE 头文件生成规则。
- 新增依赖必须同步检查 `Aura.Build.cs`。
- 必要时重新生成项目文件、刷新 IDE、重新编译或重启 UE 编辑器。
- Prompt 需要要求 Claude 报告新增文件路径、模块归属和是否需要用户手动刷新工程。

## 2026-07-07 - 正式代码必须解释设计思想

**决策：** 正式代码不仅要说明“这段代码在做什么”，还要解释关键设计为什么这样做；如果放在代码注释里会影响可读性，则补充中文 MD 文档。

**原因：** Aura 既是技能系统项目，也是 UE C++ / GAS 学习载体。用户需要理解解耦、GAS 生命周期、网络权威、表现层边界和编辑器可扩展性的设计思路。

**影响：**
- Claude Prompt 必须显式要求学习型注释和中文实现说明。
- Codex 审查时把“是否解释设计原因”作为验收点之一。

## 2026-07-13 - Phase 作为法术玩家验证角色

**决策：** 下一阶段先把 Paragon Phase 接入为玩家侧技能验证角色，完成基础移动和基础动画绑定，再继续设计新技能。

**原因：** Phase 的人物、骨骼、AnimBP、法术动作和法术风格更适合验证投射物、引导、光束、链接、范围场等技能机制。先让一个“看起来像施法角色”的验证角色跑起来，可以降低后续技能表现和动画时机验证的阻力。

**影响：**
- 当前目标不是做最终游戏角色，而是建立稳定的技能验证载体。
- 第一阶段只处理 Phase 玩家角色替换、基础 locomotion、动画蓝图接入和基础 PIE 验证。
- 暂不批量 retarget，不重构技能运行时，不批量迁移技能 DA。
- 如果 Phase 自带 Character Blueprint 和 AnimBlueprint 可用，优先复用；如果需要接入 Aura C++/GAS，则用 Aura 现有 `AAuraCharacter` 路径承载 Phase Mesh/AnimBP。
- 第三方原始资产仍保留在 `/Game/ThreePartAsset`，我们自己的试验资产放入 Aura 自己目录。

## 2026-07-13 - 视角方向采用分阶段的近第三人称准星视角

**决策：** 长期技能验证视角倾向“近一点的第三人称准星视角”，但不在 Phase 基础移动阶段一次性完成。当前 Aura 视角保留，准星视角作为 Phase 基础移动后的独立阶段。

**原因：** 准星视角更适合验证投射物、射线、光束、命中 Trace、角色朝向、摄像机朝向和多人表现。但如果在替换角色时同时改相机和输入，调试面会过大。

**影响：**
- Phase 基础接入阶段先保证角色能正常移动、转向、播放动画。
- 后续再引入屏幕中心准星、相机 yaw/pitch 控制、准星 Trace、角色朝向策略。
- 角色移动旋转策略要二选一：ARPG 移动朝向优先，或准星/控制器朝向优先；准星视角阶段再定最终参数。

## 2026-06-26 - 技能编辑器保存配置资产并提供 Montage 式预览

**决策：** 技能编辑器长期方向类比 Montage 编辑器：通过细节面板配置技能，在预览区域播放技能效果，进入 Play 后技能可实际使用；用户确认保存的技能以配置资产形式持久化，下次启动仍可使用。

**原因：** 这个类比能明确技能编辑器的交互目标：它不是简单数据表，也不是第一阶段就必须完整图编辑器，而是围绕“可配置、可预览、可运行、可保存”的技能资产工作流。

**影响：**
- 运行时模型必须能从配置资产创建可用技能。
- 编辑器后续应关注细节面板、预览场景、时序播放、资产保存和数据校验。
- 技能编辑器不生成静态 C++ 技能类。

## 提案 - 技能编辑器保存配置，不生成 C++ 类

**状态：** 提案，尚未最终确认。

**决策：** 未来技能编辑器应主要保存技能配置资产。运行时通过少量通用 C++ 能力解释和执行这些配置，而不是为每个技能生成静态 GA/GE C++ 类。

**原因：** 这更符合 UE 资产工作流，避免运行时生成 C++ 的复杂性，也更适合长期扩展。

**仍需讨论：**
- 技能行为有多少应该图形化，有多少应该结构化数据。
- 哪些部分必须保留 C++ 原生实现，以保证预测、联网和性能。
- V1 是否先用 DataAsset，图编辑器后置。
